# Quick Ledger 技术栈建议（V2）

## 1. 推荐结论

推荐使用一套“前端轻量、类型安全、易维护、适合移动端紧凑首页”的技术栈：

- 构建工具：`Vite`
- 前端框架：`React`
- 语言：`TypeScript`
- 样式方案：`CSS Modules + 原生 CSS`
- 状态管理：`React 内置状态 + 轻量自定义 store`
- 数据存储：`localStorage`
- 校验与数据建模：`zod`
- 日期处理：`date-fns`
- 测试框架：`Vitest`
- 组件测试：`Testing Library`
- 部署方式：静态站点部署

这套方案延续 V1 的轻量路线，但更适合 V2 的两个新增方向：
- 更紧凑的首页和预算面板
- iPhone 截图导入草稿的扩展流程

它比纯原生实现更易维护，比重型全家桶更简单，也足够支撑后续预算分析和自动导入能力。

---

## 2. 为什么选这套

### 2.1 简单

V2 的核心依然是本地记账，只是在首页上增加了预算表达和更紧凑的首屏信息。核心模块仍然很少：

- 记一笔
- 最近记录
- 全部记录
- 预算进度
- 本地存储

没有服务端、登录、同步、多用户权限系统，因此没必要引入重型后端或复杂全局状态方案。

### 2.2 健壮

V2 比 V1 多了预算和导入草稿，但稳定性要求不能降低：

- 金额和预算都要可靠校验
- 月度统计边界要稳定
- 本地存储要有明确封装
- 手动记账和草稿确认都要保持一致
- iPhone 截图导入必须以“草稿 + 人工确认”为前提

`TypeScript` 和 `zod` 能把这些规则收口到统一层，减少 UI 自己拼逻辑的风险。

### 2.3 适合手机端紧凑交互

V2 的首页要求在首屏同时容纳统计、记一笔、最近记录和更多入口，这种高密度布局更依赖组件化拆分。

`React` 在处理首页卡片、预算进度环、底部抽屉、最近记录预览和更多面板时，会比纯原生写法更容易保持结构清晰。

## 3. 核心技术选型

### 3.1 Vite

用途：
- 作为项目构建和开发服务器

选择理由：
- 启动快
- 配置少
- 适合单页前端项目
- 后续要加测试、PWA 或额外导入流程也方便

### 3.2 React

用途：
- 负责页面组件化
- 处理首页、统计更多面板、记账抽屉、记录列表、预算卡片等模块

选择理由：
- 组件边界清晰
- 页面壳层和弹层状态更容易管理
- 后续扩展“分类预算”“截图导入”“更多统计”更自然

### 3.3 TypeScript

用途：
- 约束记账记录、预算、草稿、页面状态和存储数据结构

选择理由：
- 记账和预算都是强结构化数据
- 避免 `amount`、`budget`、`time`、`type` 这类字段在改动时出错
- 更适合长期维护和功能迭代

### 3.4 CSS Modules + 原生 CSS

用途：
- 编写首页、统计面板、底部抽屉、按钮组、列表、进度环和预算卡片样式

选择理由：
- 简单直接
- 不依赖重型 UI 框架
- 更容易做出清爽浅绿、紧凑但有层次的视觉风格
- 便于针对手机端做精细化排版

建议：
- 使用 CSS 变量管理颜色、间距、圆角、阴影
- 使用少量过渡动画增强面板和草稿打开体验

### 3.5 localStorage

用途：
- 持久化账单、预算和草稿相关的本地数据

选择理由：
- 与当前本地优先目标完全匹配
- 不需要后端
- 浏览器关闭后仍可保留数据
- 实现成本低，适合 V2 继续保留简单结构

实现建议：
- 不要把读写散落在组件里
- 应单独封装 `storage service`
- 统一处理序列化、反序列化、异常兜底、结构校验

### 3.6 date-fns

用途：
- 处理日期格式化
- 计算今日和本月统计边界
- 计算预算展示所需的时间口径

选择理由：
- API 简单
- 体积相对小
- 比手写日期逻辑更稳

### 3.7 zod

用途：
- 进行表单、预算、草稿和存储数据校验

选择理由：
- 可以把“金额必须合法”“预算必须大于 0”“备注长度限制”“类型值必须受控”集中管理
- 与 TypeScript 配合很好
- 更适合未来增加草稿导入后的字段校验

### 3.8 Vitest + Testing Library

用途：
- 测试纯函数、存储服务、状态流和关键 UI 交互

选择理由：
- 运行快，适合前端项目
- 能覆盖 V2 里最容易出错的预算、统计和导入草稿流程
- 与 Vite 配合自然

## 4. 推荐项目结构

建议继续按功能模块组织，而不是按技术层堆文件。

```text
src/
  components/
    Button/
    StatCard/
    ProgressRing/
    RecordCard/
  features/
    ledger/
      LedgerShell.tsx
      LedgerHomeView.tsx
      LedgerRecordsView.tsx
      LedgerDrawer.tsx
      LedgerStatsPanel.tsx
      LedgerBudgetPanel.tsx
      LedgerDraftPanel.tsx
      ledger.constants.ts
      ledger.schema.ts
      ledger.form.ts
      ledger.store.ts
      ledger.stats.ts
  lib/
    storage.ts
    storage-service.ts
    date.ts
    format.ts
  styles/
    global.css
  types/
    ledger.ts
```

这样做的好处是：
- 首页和预算面板都能共享同一套状态和统计入口
- 本地存储逻辑集中
- 后续扩展截图导入或分类预算时不容易乱

---

## 5. 状态管理建议

### 5.1 不建议上重型全局状态库

V2 仍然是小型本地应用，没必要一开始就引入复杂状态方案。

### 5.2 推荐方案

使用以下方式组合即可：

- 组件本地状态管理表单输入
- 页面壳层管理当前视图、抽屉和更多面板状态
- 自定义 `ledgerStore` 负责记录、预算和草稿读写同步

如果后续预算和导入草稿的状态明显增多，再考虑更明确的状态方案也不迟。

---

## 6. 表单与校验

建议继续使用：

- 受控组件作为当前主实现
- `zod` 作为统一校验层

对于 V2 里的预算和草稿数据，也建议走同样思路：

- 手动输入
- 草稿导入后再确认
- 保存前统一校验

这样更轻，也更符合“简单但健壮”的目标。

---

## 7. 路由建议

V2 仍然不需要一上来就强依赖复杂路由系统。

建议：
- 首页
- 全部记录页
- 统计更多面板

如果最终还是单页结构，可以继续用页面状态切换，不强行上路由。

如果后续希望：
- 支持浏览器前进后退
- 支持更清晰的页面入口
- 让更多面板或截图导入有独立地址

再引入 `React Router` 也来得及。

---

## 8. iPhone 截图导入建议

V2 的截图导入不建议把 OCR 全部压进 Web 应用核心里，推荐保持“系统能力 + Web 应用承接草稿”的分层：

- iPhone 端用快捷指令 / 分享面板完成触发
- 系统能力负责截图取流和初步识别
- Web 应用只负责接收草稿、补全字段和人工确认

这样做的好处是：
- Web 应用保持轻量
- 识别流程可以逐步增强
- 即便识别失败，手动记账仍然完整可用

如果后续需要更强的浏览器内识别，再考虑接入独立 OCR 模块，但不建议作为 V2 核心依赖。

---

## 9. 测试建议

即使是小项目，也建议保留最基本的测试层。

### 9.1 单元测试

重点测：
- 金额格式化
- 时间边界计算
- 今日支出统计
- 本月支出统计
- 预算进度计算
- localStorage 读写封装

### 9.2 集成测试

重点测：
- 新增、编辑、删除流程
- 预算显示与更新流程
- 草稿确认流程

### 9.3 推荐工具

- `Vitest`
- `Testing Library`

### 9.4 测试优先级

如果时间有限，优先测纯函数和存储逻辑，因为这些地方最容易影响预算和账单正确性。

---

## 10. 部署建议

推荐继续保持静态站点部署：

- 本地开发用 Vite
- 构建产物直接部署到静态托管平台
- 不需要引入后端部署链路

这与本地优先和离线可用的产品目标一致，也方便后续继续在手机上做真实验收。
- 不需要服务器运行时

适合的部署形态：
- Vercel
- Netlify
- GitHub Pages
- 任意静态资源托管

---

## 10. 不建议使用的方案

### 10.1 不建议纯原生作为最终方案

纯原生 HTML/CSS/JS 当然能做，但在后续扩展编辑、删除、抽屉交互、状态同步、数据封装时，维护成本会更快上升。

### 10.2 不建议上重型全栈框架

不需要：
- 服务端渲染
- 数据库
- 登录系统
- API 层

这些都会让 V1 变重。

### 10.3 不建议一开始引入过多第三方 UI 库

这个产品需要的是：
- 快
- 轻
- 清爽

大 UI 库会让体积、样式控制和定制成本都变高。

---

## 11. 最终建议

如果只选一套最适合当前阶段的技术栈，我建议：

- `Vite`
- `React`
- `TypeScript`
- `CSS Modules + 原生 CSS`
- `zod`
- `date-fns`
- `localStorage`

这套组合在“简单”和“健壮”之间比较平衡，足以支撑 V1，而且未来扩展收入统计、快捷记账、PWA 安装体验时也不会推倒重来。
