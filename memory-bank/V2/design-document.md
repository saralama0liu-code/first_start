# Quick Ledger 产品设计文档（V2）

## 1. 文档概述

### 1.1 产品名称
Quick Ledger（超快记账工具）

### 1.2 文档目的
本文档描述 Quick Ledger 的 V2 设计方案。V1 已作为稳定基线封存，本版面向“更紧凑的手机首页 + 月度预算可视化 + iPhone 截图导入草稿”三个核心方向，作为后续开发与验收依据。

### 1.3 V2 设计原则
- 极简，但比 V1 更有信息层次
- 首页首屏更紧凑，重要信息更突出
- 绿色、清爽、轻量
- 本地优先，不依赖云端
- 保持手动记账能力完整可用
- 自动导入只能作为“辅助草稿”，不直接替代确认

---

## 2. 产品目标

### 2.1 核心目标
- 让用户打开手机后，在一个页面内快速看见统计、预算和最近记录
- 让用户在支付完成后，通过 iPhone 截图导入更快生成记账草稿
- 让用户理解“实际消费”和“计划消费”的差距
- 让首页比 V1 更紧凑、更清爽、更像一个消费管理仪表盘

### 2.2 V2 成功标准
- 手机首屏可以同时看到统计、记一笔入口、最近记录预览和更多入口
- 统计区能清楚展示本月实际消费与计划消费的关系
- 用户可以通过 iPhone 截图导入生成待确认草稿
- 手动新增、编辑、删除依然保持完整
- 同一设备同一浏览器内的数据持久化继续可靠

---

## 3. 产品范围

### 3.1 V2 包含功能
- 首页紧凑化重设计
- 清爽浅绿视觉风格
- 月度总预算展示
- 分类预算与实际支出对比
- 最近记录小卡片预览
- 更多入口收纳“查看全部记录”和预算管理
- iPhone 截图导入草稿流程
- 新增、编辑、删除记账记录
- 本地持久化保存

### 3.2 V2 暂不包含功能
- 云端同步
- 登录 / 注册
- 自动直连微信支付或支付宝支付结果
- 多设备同步
- 复杂报表和趋势图
- PWA 桌面安装
- 收支自动识别后的直接入账

### 3.3 后续预留能力
- 更细的消费分析图表
- 更强的自动识别规则
- 更多预算拆分方式
- 智能推荐常用分类或金额

## 4. 目标用户与场景

### 4.1 目标用户
- 需要快速记账的手机用户
- 经常使用微信支付或支付宝支付的用户
- 想看预算执行情况但不想打开复杂 App 的用户
- 希望减少操作步骤、减少文字干扰的用户

### 4.2 典型场景
- 支付完成后，用户直接截图并通过快捷指令导入草稿
- 用户打开首页后，先看本月预算进度，再快速记一笔
- 用户只想在首页看到最重要的统计和最近记录，不想被长文案打扰
- 用户需要进入“更多”查看全部记录或编辑预算

---

## 5. 需求确认结论

以下是当前确认的 V2 决策：

1. 截图导入主流程采用 iPhone 快捷指令 + 截图识别 + 草稿确认。
2. 自动识别优先提取金额、商户名和时间，分类先由用户确认。
3. 识别结果不直接自动入账，先生成可编辑草稿。
4. 首页必须保持单屏紧凑，顶部统计和记一笔入口更突出。
5. 最近记录只显示 2 到 4 条，尽量压缩高度。
6. “查看全部记录”保留，但收进更多入口。
7. 整体视觉改成清爽浅绿风格。
8. 首页长解释文案大幅减少，只保留必要信息。
9. 手动新增、编辑、删除继续保留。
10. 预算按月计算。
11. 月度总预算默认提供一个基础值，并显示未设置或引导提示。
12. 分类预算也纳入 V2，但放在统计页的更多区域里，不挤压首页。
13. 分类计划和实际支出对比是 V2 的一部分。
14. 数据继续保持本地优先，不做云同步。

## 7. 页面设计

## 7.1 首页

### 7.1.1 页面目标
首页承担“总览 + 快速记一笔 + 快速浏览最近记录”的职责，并且必须在手机首屏尽量完成信息呈现。

### 7.1.2 视觉方向
- 浅色背景
- 清爽浅绿主色
- 卡片层级清晰
- 留白更少，但信息密度更高
- 文字更短，操作更直接

### 7.1.3 页面组成
- 顶部统计区
- 主要操作区
- 最近记录预览区
- 更多入口

### 7.1.4 顶部统计区
展示内容：
- 今日支出
- 本月支出
- 本月预算进度
- 剩余可用预算或超支状态

展示规则：
- 仅以支出为核心统计
- 预算采用月度口径
- 预算卡片以进度环或进度弧形式展示
- 当没有设置预算时，展示默认基础值和引导提示

### 7.1.5 主要操作区
主要入口：
- “记一笔”主按钮

交互要求：
- 按钮明显、面积大、视觉优先级最高
- 点击后打开底部记账面板

### 7.1.6 最近记录预览区
展示数量：
- 默认显示 2 到 4 条

展示字段：
- 金额
- 分类或商户简写
- 时间
- 可选备注摘要

交互：
- 点击单条记录进入编辑
- 每条记录保留一个更多按钮
- 更多按钮里可触发编辑或删除
- 预览区顶部保留“更多”入口，打开全部记录和预算相关内容

### 7.1.7 更多入口
更多入口建议包含：
- 查看全部记录
- 预算管理
- 分类预算管理
- 截图导入说明

## 7.2 统计更多面板

### 7.2.1 页面目标
统计更多面板用于查看预算执行情况、分类计划与实际支出的关系，并承担更深一层的预算设置入口。

### 7.2.2 展示内容
- 月度总预算
- 月度实际支出
- 本月剩余预算或超支数值
- 分类预算列表
- 各分类实际支出
- 各分类计划占比

### 7.2.3 视觉表达
- 总预算使用更强的进度可视化
- 分类预算使用列表式进度条或迷你环形展示
- 重点突出“计划 vs 实际”的比例

### 7.2.4 交互能力
- 点击总预算可进入修改
- 点击分类条目可修改该分类计划
- 支持从“更多”进入全部记录页

### 7.2.5 默认值策略
- 月度总预算提供默认基础值
- 如果用户未设置，显示引导提示
- 分类预算默认可为空，或先使用未设置状态提示

## 7.3 记账面板

### 7.3.1 页面目标
记账面板仍然承担“快速新增 / 编辑记录”的核心职责。

### 7.3.2 展示方式
- 从底部滑出
- 移动端接近全屏
- 保持首页上下文可感知，但主操作聚焦在面板内

### 7.3.3 面板内容
- 顶部标题与关闭按钮
- 类型切换
- 金额输入区
- 分类选择
- 支付方式选择
- 时间选择
- 备注输入
- 保存按钮

### 7.3.4 类型切换
- 支出
- 收入

默认值：
- 支出

说明：
- 首页核心统计仍以支出为主
- 收入记录保留，但不进入首页主视觉统计

### 7.3.5 金额输入区
设计要求：
- 金额区域视觉优先级最高
- 使用系统数字键盘输入
- 显示清晰的大字号金额

规则：
- 金额必填
- 金额必须大于 0
- 允许小数
- 最多两位小数

### 7.3.6 分类选择
固定分类：
- 餐饮
- 交通
- 日用
- 娱乐
- 学习
- 其他

交互：
- 使用按钮式选择
- 单选
- 新增时可记住上次使用项
- 编辑时使用记录原值

### 7.3.7 支付方式选择
固定支付方式：
- 微信
- 支付宝
- 银行卡
- 现金

交互：
- 使用按钮式选择
- 单选
- 新增时可记住上次使用项
- 编辑时使用记录原值

### 7.3.8 时间选择
规则：
- 默认自动记录当前时间
- 允许用户手动修改

交互建议：
- 默认以当前时间展示
- 只有点击修改时才展开编辑

### 7.3.9 备注输入
规则：
- 可选填写
- 长度限制 50 个字符

目的：
- 避免备注过长破坏列表可读性
- 满足补充说明的基本需求

### 7.3.10 保存行为
保存成功后：
- 生成新记录或更新原记录
- 新增后清空当前表单并保持面板打开
- 编辑后关闭面板并返回列表
- 给出成功反馈，例如“已保存”

保存失败时：
- 保持当前内容
- 提示未通过校验的原因

## 7.5 截图导入草稿面板

### 7.5.1 页面目标
让 iPhone 用户在支付完成后，通过截图导入生成一条待确认草稿，减少手动输入量。

### 7.5.2 主流程
1. 用户在微信或支付宝支付完成后截图
2. 用户通过 iPhone 分享面板运行快捷指令
3. 快捷指令识别截图中的金额、商户名和时间
4. 快捷指令把识别结果带回应用
5. 应用生成可编辑草稿
6. 用户确认后保存

### 7.5.3 设计要求
- 草稿不是最终入账结果
- 识别结果必须允许人工修正
- OCR 置信度低时要明确提示用户补充
- 图片不作为长期存储内容

### 7.5.4 失败兜底
- 如果识别失败，仍然允许回到手动记账面板
- 如果快捷指令不可用，页面应清楚说明手动方式仍可使用

## 8. 核心交互流程

### 8.1 新增记账流程
1. 用户点击首页“记一笔”
2. 底部记账面板弹出
3. 用户输入金额
4. 用户选择分类和支付方式
5. 如有需要，用户修改时间并填写备注
6. 点击保存
7. 成功后表单清空并保留在面板内

### 8.2 编辑记账流程
1. 用户点击某条记录
2. 打开记账面板并预填记录内容
3. 用户修改字段
4. 点击保存
5. 更新记录并关闭面板

### 8.3 删除记账流程
1. 用户在列表中触发删除
2. 系统确认是否删除
3. 用户确认后删除记录
4. 列表、预算统计和首页统计同步刷新

### 8.4 预算查看流程
1. 用户打开首页
2. 系统自动计算本月实际消费和预算进度
3. 用户点击更多打开统计面板
4. 系统展示总预算与分类计划/实际支出

### 8.5 截图导入流程
1. 用户支付完成后截图
2. 用户通过 iPhone 快捷指令导入截图
3. 系统识别金额、时间和商户
4. 用户确认草稿
5. 保存为正式记录

## 9. 数据设计

### 9.1 数据存储方案
- 使用浏览器 `localStorage`
- 适合轻量、离线、本地优先场景

### 9.2 数据边界
仅保证：
- 同一设备
- 同一浏览器
- 正常刷新或关闭后再次打开仍可恢复

不保证：
- 清理浏览器缓存后仍存在
- 跨设备同步
- 跨浏览器同步

### 9.3 数据模型

#### 9.3.1 记账记录
```json
[
  {
    "id": "唯一ID",
    "type": "expense",
    "amount": 25.5,
    "category": "餐饮",
    "payment": "微信",
    "time": "2026-04-22T12:30:00.000Z",
    "note": "午饭"
  }
]
```

#### 9.3.2 预算设置
```json
{
  "monthlyBudget": 2000,
  "categoryBudgets": {
    "餐饮": 600,
    "交通": 300
  }
}
```

#### 9.3.3 草稿数据
```json
{
  "amount": 18.5,
  "merchant": "便利店",
  "time": "2026-04-22T12:30:00.000Z",
  "category": "日用"
}
```

### 9.4 字段说明
- `id`：唯一标识
- `type`：`expense` 或 `income`
- `amount`：金额，数字类型
- `category`：分类名称
- `payment`：支付方式名称
- `time`：ISO 8601 时间字符串
- `note`：备注文本，可空
- `monthlyBudget`：月度总预算
- `categoryBudgets`：分类预算映射

## 10. 统计规则

### 10.1 今日支出
- 统计当天所有 `expense` 类型记录的金额总和
- 时间按本地时区判断当天边界

### 10.2 本月支出
- 统计当月所有 `expense` 类型记录的金额总和
- 时间按本地时区判断月边界

### 10.3 预算进度
- 预算进度 = 本月实际支出 / 本月预算
- 进度超过 100% 时显示超支状态
- 未设置预算时显示默认值和提示

### 10.4 分类预算统计
- 统计每个分类的本月支出
- 对比该分类的计划额度
- 展示计划和实际占比

### 10.5 统计展示规则
- 仅展示支出统计
- 收入不进入首页核心统计
- 金额统一保留一致的货币格式

## 11. 校验规则

### 11.1 必填项校验
- 金额必填
- 分类必选
- 支付方式必选
- 时间必须有值，默认自动生成

### 11.2 格式校验
- 金额必须为有效数字
- 金额必须大于 0
- 金额最多两位小数
- 备注长度不超过 50 个字符
- 预算值必须为有效数字且大于 0

### 11.3 交互反馈
- 保存成功后提示“已保存”
- 删除前必须二次确认
- 校验失败时在对应字段附近提示原因
- OCR 识别结果不确定时提示用户人工确认

---

## 12. 交互与视觉要求

### 12.1 交互要求
- 所有操作尽量减少输入
- 优先按钮选择，少用键盘输入
- 关键按钮尺寸足够大
- 适合单手操作
- 统计卡片和记一笔入口应优先于长说明文字

### 12.2 视觉要求
- 首页和统计面板层级清晰
- 金额区域最醒目
- 列表信息简洁可扫读
- 采用清爽浅绿色主题
- 减少深色压迫感，保持明亮和轻快

---

## 13. 非功能性要求

- 页面加载快速
- 无网络也可使用核心功能
- 手机端适配优先
- iPhone Safari 与常用移动浏览器可正常使用
- 代码结构清晰，便于未来扩展
- 快捷指令导入属于增强能力，不能破坏基础记账可用性

---

## 14. 异常与边界情况

### 14.1 数据为空
- 首页显示空状态
- 统计显示 0
- 最近记录列表显示轻量提示

### 14.2 金额非法
- 不允许保存
- 提示用户修正金额

### 14.3 删除误操作
- 删除前二次确认
- 删除后列表和统计立即刷新

### 14.4 本地存储异常
- 若 `localStorage` 不可用，应给出明确提示
- 不做临时内存降级

### 14.5 截图识别失败
- 识别失败时保留手动记账入口
- 不强制用户重新截图
- 明确告诉用户哪些字段没有识别成功
