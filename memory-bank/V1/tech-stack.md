# Quick Ledger 技术栈建议

## 1. 推荐结论

推荐使用一套“静态站点优先、前端轻量、类型安全、易维护”的技术栈：

- 构建工具：`Vite`
- 前端框架：`React`
- 语言：`TypeScript`
- 样式方案：`CSS Modules + 原生 CSS`
- 状态管理：`React 内置状态 + 轻量自定义 store`
- 数据存储：`localStorage`
- 工具库：`date-fns`、`zod`
- 部署方式：静态站点部署

这套方案比纯原生实现更易维护，比重型全家桶更简单，比较适合当前这个“极简但要稳”的记账产品。

---

## 2. 为什么选这套

### 2.1 简单

这个产品的核心只有几个模块：

- 记一笔
- 列表
- 统计
- 本地存储

不需要服务端、不需要登录、不需要复杂权限系统，因此没必要上后端框架或重型状态方案。

### 2.2 健壮

虽然功能少，但有几个地方不能太随意：

- 金额校验要可靠
- 时间处理要稳定
- 本地存储要有明确封装
- 编辑、删除、统计要保持一致

`TypeScript` 和 `zod` 能明显提高这部分的可靠性。

### 2.3 适合手机端快速交互

底部滑出式记账面板、本地统计、快捷按钮和列表刷新，都是前端交互密集型场景。

`React` 在处理组件拆分、表单状态、弹层状态、列表渲染时，会比纯原生写法更容易保持结构清晰。

---

## 3. 核心技术选型

### 3.1 Vite

用途：
- 作为项目构建和开发服务器

选择理由：
- 启动快
- 配置少
- 适合纯前端小型项目
- 后续要加 PWA、路由或测试也很方便

### 3.2 React

用途：
- 负责页面组件化
- 处理首页、记账面板、全部记录页、列表项、统计区等模块

选择理由：
- 组件边界清晰
- 弹层和表单状态更容易管理
- 后续扩展“收入统计”“重复上一笔”“桌面安装”更自然

### 3.3 TypeScript

用途：
- 约束数据结构
- 约束表单字段
- 约束本地存储读取和写入

选择理由：
- 记账数据是强结构化数据
- 避免 `amount`、`time`、`type` 这类字段在改动时出错
- 后续加统计功能时更不容易引入隐性 bug

### 3.4 CSS Modules + 原生 CSS

用途：
- 编写首页、抽屉面板、按钮组、列表、统计卡片样式

选择理由：
- 简单直接
- 不依赖重型 UI 框架
- 更容易做出“极简但有辨识度”的视觉风格
- 便于针对手机端做细粒度控制

建议：
- 使用 CSS 变量管理颜色、间距、圆角、阴影
- 使用少量过渡动画提升面板打开和关闭体验

### 3.5 localStorage

用途：
- 持久化账单数据

选择理由：
- 与当前 V1 目标完全匹配
- 不需要后端
- 浏览器关闭后仍可保留数据
- 实现成本极低

实现建议：
- 不要把读写散落在组件里
- 应单独封装 `storage service`
- 统一处理序列化、反序列化、异常兜底、版本迁移

### 3.6 date-fns

用途：
- 处理日期格式化
- 计算今日和本月统计边界

选择理由：
- API 简单
- 体积相对小
- 比手写日期逻辑更稳

### 3.7 zod

用途：
- 进行表单和存储数据校验

选择理由：
- 可以把“金额必须合法”“备注长度限制”“类型值必须受控”这类规则集中管理
- 与 TypeScript 配合非常好

## 4. 推荐项目结构

建议按功能模块组织，而不是按技术层堆文件。

```text
src/
  components/
    CommonButton/
    RecordCard/
    StatCard/
  features/
    ledger/
      LedgerPage.tsx
      RecordDrawer.tsx
      RecordList.tsx
      RecordForm.tsx
      stats.ts
      ledgerStore.ts
      ledgerSchema.ts
  lib/
    storage.ts
    date.ts
    format.ts
  styles/
    variables.css
    global.css
  types/
    ledger.ts
```

这样做的好处是：
- 页面和业务逻辑分离
- 本地存储逻辑集中
- 后续扩展收入统计或安装体验时不容易乱

---

## 5. 状态管理建议

### 5.1 不建议上重型全局状态库

这个产品规模不大，没必要一开始就引入复杂状态方案。

### 5.2 推荐方案

使用以下方式组合即可：

- 组件本地状态管理表单输入
- 父组件管理页面级状态
- 自定义 `ledgerStore` 负责记录读写和同步

如果后续页面明显增多，再考虑引入更明确的全局状态方案也不迟。

---

## 6. 表单与校验

建议使用：

- `React Hook Form`，如果后续表单复杂度上升
- 或直接使用受控组件 + `zod` 校验，当前阶段也足够

当前 V1 的字段不多，我更倾向于：

- 先用受控组件
- 配合 `zod` 做提交前校验

这样更轻，也更符合“简单但健壮”的目标。

---

## 7. 路由建议

当前 V1 可以不强依赖复杂路由系统。

建议：
- 首页
- 全部记录页

如果项目最终仍是单页面结构，甚至可以只用组件状态切换，不强行上路由。

如果后续希望：
- 支持浏览器前进后退
- 支持更清晰的页面入口
- 支持未来安装成 PWA 后的导航体验

再引入 `React Router` 也来得及。

---

## 8. 测试建议

即使是小项目，也建议保留最基本的测试层。

### 8.1 单元测试

重点测：
- 金额格式化
- 时间边界计算
- 今日支出统计
- 本月支出统计
- localStorage 读写封装

### 8.2 推荐工具

- `Vitest`
- `Testing Library`

### 8.3 测试优先级

如果时间有限，优先测纯函数和存储逻辑，因为这些地方最容易影响账单正确性。

---

## 9. 部署建议

推荐直接部署为静态站点：

- 本地开发用 Vite
- 构建产物直接部署到静态托管平台
- 不需要服务器运行时

适合的部署形态：
- Vercel
- Netlify
- GitHub Pages
- 任意静态资源托管

---

## 10. 不建议使用的方案

### 10.1 不建议纯原生作为最终方案

纯原生 HTML/CSS/JS 当然能做，但在后续扩展编辑、删除、抽屉交互、状态同步、数据封装时，维护成本会更快上升。

### 10.2 不建议上重型全栈框架

不需要：
- 服务端渲染
- 数据库
- 登录系统
- API 层

这些都会让 V1 变重。

### 10.3 不建议一开始引入过多第三方 UI 库

这个产品需要的是：
- 快
- 轻
- 清爽

大 UI 库会让体积、样式控制和定制成本都变高。

---

## 11. 最终建议

如果只选一套最适合当前阶段的技术栈，我建议：

- `Vite`
- `React`
- `TypeScript`
- `CSS Modules + 原生 CSS`
- `zod`
- `date-fns`
- `localStorage`

这套组合在“简单”和“健壮”之间比较平衡，足以支撑 V1，而且未来扩展收入统计、快捷记账、PWA 安装体验时也不会推倒重来。
