# Quick Ledger 实施计划

## 1. 文档目的

本文档用于指导 AI 开发者按小步、可验证的方式实现 Quick Ledger V1。  
要求如下：

- 每一步必须足够小，目标单一明确
- 每一步必须包含可执行的验证方式
- 每一步只描述实现指令，不包含代码
- 每一步完成后再进入下一步

---

## 2. 实施原则

- 严格以 [design-document.md](</Users/liuxuan/Documents/VIBE%20CODING/记账/memory-bank/V1/design-document.md>) 和 [tech-stack.md](</Users/liuxuan/Documents/VIBE%20CODING/记账/memory-bank/V1/tech-stack.md>) 为准
- 使用 `Vite + React + TypeScript + CSS Modules + localStorage + zod + date-fns`
- 保持移动端优先
- 不提前实现 V1 范围外功能
- 所有数据读写必须通过统一封装，不允许在多个 UI 组件中直接散落读写
- 所有核心逻辑都必须可被测试或验证

---

## 3. AI 开发者执行方式

每一步提交时，输出内容应包含：

1. 本步完成了什么
2. 修改了哪些文件
3. 如何运行验证
4. 验证结果是否通过
5. 若未通过，阻塞点是什么

---

## 4. 已确认的关键决策

以下规则已确认，实施时不得再自行改写：

- `全部记录` 使用同一页里的视图切换，不做独立路由页面
- 新增保存后清空表单并停留在抽屉内
- 编辑保存后关闭抽屉并返回列表
- 分类和支付方式在新增时记住上次选择，在编辑时使用记录原值
- 时间字段默认显示当前时间，只有点击修改时才展开编辑
- 删除入口使用“更多菜单里删除 + 二次确认”
- 首页和全部记录页都支持删除
- 备注最多 50 个字符
- 金额必须大于 0，允许小数，最多两位小数，不设上限
- 时间存储使用 ISO 8601，展示时再格式化
- `localStorage` 不可用时直接提示无法使用，不做临时内存降级
- 空状态与反馈文案需统一
- 验收要求为核心逻辑自动化测试 + 主流程手测

- 抽屉达到可交付体验

## Step 20: 处理空状态、异常状态与降级场景

### 指令
补齐所有关键边界状态，至少包括：

- 初次使用无记录
- 统计为 0
- 存储读取失败
- 存储数据损坏
- 表单提交失败
- 本地存储不可用

确保这些状态下页面依然可理解、可恢复、不崩溃。

### 交付要求
- 每类状态都有明确反馈
- 页面不会因为异常数据直接崩溃
- 异常不会静默吞掉

### 验证测试
- 清空存储后首次打开页面，确认空状态正常
- 手动污染本地存储数据，确认页面仍可打开
- 模拟存储服务返回失败，确认用户可见错误提示
- 模拟 `localStorage` 不可用，确认直接提示无法使用

### 完成标准
- 产品具备基本健壮性

## Step 21: 完成单元测试与关键交互测试

### 指令

- 时间工具函数

- 本地存储服务
- 新增记录流程

### 交付要求

- 主要用户流程有交互级验证
- 测试命名清晰，可维护

### 验证测试

- 人工检查是否覆盖“收入不计入首页支出统计”这一核心规则
- 人工检查是否覆盖“编辑后刷新统计”这一关键流程

## 6. 建议执行顺序

## 8. 额外约束
